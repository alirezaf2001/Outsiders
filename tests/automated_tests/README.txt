AUTOMATED TESTS README
======================

This folder contains simple runnable smoke tests for the Swing + MySQL
employee management system.

Structure
---------
- run_all.sh
    Compiles the app and the automated test classes, then runs the test suite.

- java/automated_tests/*.java
    Standalone Java test programs. No JUnit is required.

- results/*.csv
    Spreadsheet-ready output generated by each run. One file contains the
    detailed PASS / FAIL checks, and one file contains suite-level summary rows.

What is covered
---------------
UI smoke / validation tests:
- app frame launches
- login panel field reset
- add-employee form validation
- salary-update form validation
- employee-search validation
- reports-panel validation

Database smoke tests:
- database connection opens
- employee / payroll / report service reads work
- add / update / delete round-trip for a temporary employee

Database cleanup rule
---------------------
The round-trip database test creates a temporary employee with a unique email
address and deletes that employee in cleanup. It does not modify permanent test
data on purpose.

What is intentionally not automated here
----------------------------------------
- live salary updates against existing employees
- destructive edits against permanent employee rows

Those cases stay in manual testing because they change real database data and
would require a fuller rollback strategy.

How to run
----------
From the project root:

  ./tests/automated_tests/run_all.sh

If MySQL is not running and you only want the Swing/UI smoke tests:

  OUTSIDERS_SKIP_DB=1 ./tests/automated_tests/run_all.sh

After each run
--------------
The runner exports two CSV files into tests/automated_tests/results:

- automated_test_results_<timestamp>.csv
    One row per individual PASS / FAIL / INFO check

- automated_test_suite_summary_<timestamp>.csv
    One row per suite with passed / failed counts

It also refreshes:

- automated_test_results_latest.csv
- automated_test_suite_summary_latest.csv

These files open directly in Excel or Google Sheets.

Notes
-----
- The UI tests open real Swing windows briefly.
- The DB tests use the credentials in src/db/DatabaseConfig.java.
- If a DB test fails, read the console output first; it is meant to be direct
  enough to diagnose the problem quickly.
